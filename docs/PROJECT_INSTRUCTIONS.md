# AI RADAR — ПРОЕКТНЫЕ ИНСТРУКЦИИ

## 1. Общие принципы

AI Radar — модульный конвейер обработки новостей об ИИ.

Каждый новый этап должен сохранять существующую архитектуру, не нарушать разделение ответственности, быть покрыт тестами и не изменять предыдущие этапы без необходимости. Любое изменение должно быть минимальным.

## 2. Архитектурные границы

Запрещается смешивать Collection, Normalize, Analysis, Validation, Moderation, Human Review, Publication и Operations.

Pipeline только оркестрирует. Scheduler только запускает Pipeline. Publication публикует. Operations наблюдает.

## 3. Восстановление контекста

Перед началом работы обязательно изучить:

- `STATE.md`;
- `LOG.md`;
- последнюю фазу;
- последний отчёт.

Без восстановления контекста работа не начинается.

## 4. Выполнение задач

Каждая задача оформляется отдельной папкой:

`/storage/emulated/0/Atlas Lab/01_Context/YYYY-MM-DD_<TASK>/`

В ней обязательны TASK, REPORT, дополнительные материалы, pytest, git diff, логи и примеры CLI.

## 5. Ограничения

По умолчанию запрещены удаление данных, destructive SQL, изменение секретов, системных сервисов, CI/CD и Docker без необходимости, а также изменение прошлых этапов.

## 6. Git

Работа выполняется в отдельной ветке. После завершения предоставить Commit SHA, список файлов, git diff и результаты тестов. Merge не выполнять.

## 7. База данных

Изменения схемы допускаются только через Alembic. Каждая миграция должна поддерживать upgrade и downgrade и проходить проверку на чистой БД.

## 8. Тестирование

Каждый этап обязан включать новые unit-тесты, прохождение полного существующего набора тестов и отсутствие регрессий.

## 9. Отчёт

Отчёт должен содержать реализованное, созданные и изменённые файлы, результаты pytest, примеры CLI, Commit SHA, ограничения, риски и итоговый статус.

Базовый статус разработчика:

```text
SUCCESS
Needs Review = true
Verified = false
```

## 10. Запрещённые интеграции

До отдельного задания запрещены Telegram, Discord, Email, REST API, Web UI, Prometheus, Grafana и внешние интеграции.

## 11. Приоритет разработки

1. Архитектура.
2. Надёжность.
3. Тестируемость.
4. Наблюдаемость.
5. Производительность.
6. Пользовательские интерфейсы.

## 12. Критерии приёмки

Этап завершён только если архитектура не нарушена, все тесты проходят, отсутствуют побочные эффекты, отчёт сформирован, изменения документированы и `STATE.md` обновлён.
